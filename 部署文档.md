# 文档智能系统部署说明

## 1.部署架构

推荐使用Docker Compose进行单服务器部署：

- `frontend`：Nginx提供Vue生产静态文件，并将`/api`请求反向代理到后端。
- `backend`：FastAPI与Uvicorn，固定使用单worker运行。
- `postgres`：PostgreSQL 16，保存用户、会话、文档库、工作流和执行状态。
- `app_data`：保存上传文件、生成文件、缓存和日志。
- `postgres_data`：保存数据库数据。

工作流任务当前由后端进程内线程执行，因此不要把Uvicorn worker数量改为多个，也不要同时启动多个backend副本。正在运行任务时重启backend会中断该任务。

## 2.服务器要求

- Linux x86_64服务器，推荐Ubuntu 22.04或24.04。
- Docker Engine 24+与Docker Compose v2。
- 至少4核CPU、8GB内存、20GB可用磁盘。
- 开放应用端口，默认是`8080`。
- 服务器可以访问所配置的LLM服务，例如`api.deepseek.com:443`。

首次构建会安装文档解析、PDF和AI相关Python依赖，镜像较大，构建时间可能较长。
后端镜像使用`requirements-deploy.txt`中的已验证固定版本构建，开发环境仍可使用根目录的`requirements.txt`。

## 3.准备部署文件

解压部署包并进入目录：

```bash
unzip Document-Intelligence-System.zip
cd Document-Intelligence-System
```

复制生产环境变量模板：

```bash
cp .env.production.example .env.production
```

生成认证密钥：

```bash
python3 -c "import secrets; print(secrets.token_urlsafe(48))"
```

编辑`.env.production`，至少修改以下配置：

```dotenv
POSTGRES_PASSWORD=高强度数据库密码
AUTH_SECRET_KEY=上一步生成的随机字符串
LLM_PROVIDER=deepseek
LLM_MODEL=deepseek-chat
DEEPSEEK_API_KEY=实际API密钥
```

不要把`.env.production`提交到Git，也不要发送给其他人。

## 4.检查LLM网络

DeepSeek示例：

```bash
curl -I --connect-timeout 10 https://api.deepseek.com
```

只要能够建立HTTPS连接即可，未携带密钥时返回非200状态也属正常。如果出现`Could not connect to server`，需要检查服务器防火墙、云安全组、代理或网络出口。

使用代理时，在`.env.production`中配置：

```dotenv
HTTPS_PROXY=http://代理地址:端口
HTTP_PROXY=http://代理地址:端口
```

Docker部署在Windows或macOS且代理运行在宿主机时，可尝试：

```dotenv
HTTPS_PROXY=http://host.docker.internal:7890
HTTP_PROXY=http://host.docker.internal:7890
```

## 5.启动系统

先检查Compose配置：

```bash
docker compose --env-file .env.production config
```

构建并启动：

```bash
docker compose --env-file .env.production up -d --build
```

查看容器状态：

```bash
docker compose --env-file .env.production ps
```

查看启动日志：

```bash
docker compose --env-file .env.production logs -f backend
docker compose --env-file .env.production logs -f frontend
docker compose --env-file .env.production logs -f postgres
```

## 6.部署验证

检查前端：

```bash
curl http://127.0.0.1:8080/healthz
```

预期返回：

```text
ok
```

检查后端和数据库：

```bash
curl http://127.0.0.1:8080/health
curl http://127.0.0.1:8080/ready
```

预期结果中应包含：

```json
{
  "success": true,
  "data": {
    "status": "ok",
    "database_enabled": true,
    "database_ok": true
  }
}
```

`/ready`应返回`{"success":true,"data":{"status":"ready"}}`。数据库不可连接时，该接口返回HTTP 503，backend容器也会进入不健康状态。

浏览器访问：

```text
http://服务器IP:8080
```

首次进入后注册用户并登录，然后依次验证：

1. 文档库上传与下载文件。
2. 智能对话选择文档并发起提问。
3. 文档编辑生成并下载输出文件。
4. 工作流执行并观察节点进度。

## 7.数据库初始化说明

PostgreSQL容器首次创建数据卷时，会按顺序执行：

1. `sql/schema_v1.sql`
2. `sql/001_create_sessions_tables.sql`
3. `sql/002_auth_user_file_scope.sql`
4. `sql/003_library_tables.sql`
5. `sql/004_workflow_tables.sql`

初始化脚本只会在`postgres_data`为空时自动执行。已有数据库升级时，应先备份，再手动执行新增SQL脚本。

## 8.备份与恢复

备份数据库：

```bash
docker compose --env-file .env.production exec -T postgres \
  pg_dump -U docintel -d doc_intel > doc_intel_backup.sql
```

恢复数据库：

```bash
cat doc_intel_backup.sql | docker compose --env-file .env.production exec -T postgres \
  psql -U docintel -d doc_intel
```

查看数据卷：

```bash
docker volume ls
```

生产环境还需要定期备份`app_data`卷，其中保存用户上传文件和生成结果。

## 9.更新部署

更新代码或替换部署包后执行：

```bash
docker compose --env-file .env.production build
docker compose --env-file .env.production up -d
```

更新前确认没有正在运行的工作流任务，并先备份数据库与`app_data`卷。

## 10.停止与卸载

停止服务但保留数据：

```bash
docker compose --env-file .env.production down
```

删除服务和全部数据卷：

```bash
docker compose --env-file .env.production down -v
```

`down -v`会永久删除数据库和本地文件，执行前必须确认已经备份。

## 11.常见问题

### LLM显示Connection error

进入backend容器检查网络：

```bash
docker compose --env-file .env.production exec backend \
  python -c "import urllib.request; print(urllib.request.urlopen('https://api.deepseek.com', timeout=10).status)"
```

如果无法连接，检查代理、防火墙和服务器出口网络。如果返回401或402，再检查API密钥、账户余额和模型权限。

### 前端访问接口返回502

检查backend是否健康：

```bash
docker compose --env-file .env.production ps
docker compose --env-file .env.production logs --tail=200 backend
```

### 数据库初始化失败

查看PostgreSQL日志：

```bash
docker compose --env-file .env.production logs --tail=300 postgres
```

如果是首次部署且数据库内没有需要保留的数据，可以执行`down -v`后重新初始化。

### 上传大文件失败

Nginx当前限制为100MB，后端默认业务限制为50MB。如需调整，应同步修改`deploy/nginx.conf`中的`client_max_body_size`和后端文件大小配置。

## 12.当前部署边界

- 后端必须保持单worker、单副本。
- 工作流任务不是外部队列任务，backend重启会中断正在执行的任务。
- LLM能力依赖外部服务网络、密钥、余额和服务可用性。
- 建议先在内网或答辩服务器部署；如果需要公网高可用，应进一步引入Celery/RQ等任务队列、Redis、独立worker和完整的工作流用户隔离审计。
